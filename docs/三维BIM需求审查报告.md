# 三维 BIM 需求审查报告

审查时间：2026-04-24

审查范围：

- `/visualization/3d-fullscreen`
- `src/views/visualization/viewer3d`
- `docs/三维BIM借鉴广联达功能建议.md`
- `docs/广联达BIM功能调研.md`
- `docs/三维全屏场景需求符合性核查清单.md`
- `docs/三维全屏场景整改实施计划.md`
- `docs/三维全屏场景整改任务排期表.md`

## 一、审查结论

当前三维 BIM 需求方向成立，但需求文档需要收敛。现阶段不建议继续扩展 4D、5D、BIM+GIS、AI 助理等大范围能力，应先完成“可验收的三维交付底座”。

当前最大问题不是三维功能不足，而是以下四类问题：

- 需求口径偏大：存在“全厂级”“实时高清”“定期扫描更新”等宽泛描述。
- 验收标准不足：缺少可量化指标、资产清单、数据来源、性能指标。
- 文档与代码状态不完全一致：部分文档仍写“缺少导航图、缺少台账参数”，但当前代码中已存在导航图组件和设备台账/运行参数展示。
- 外部依赖未拆清：全厂资产覆盖、真实实时参数、扫描建模更新、4D/5D 数据都依赖模型侧、业务侧或后台接口，不应只归为前端需求。

建议当前需求重新定义为：

`完成三维 BIM 可视化交付底座：支持模型资产管理、三维浏览、结构树、导航定位、构件属性、KKS/文档/设备绑定、测量剖切、点位摄像头、场景保存、基础实时数据展示，并提供可验收的资产覆盖清单和数据来源说明。`

| 需求项 | 当前判断 | 处理建议 |
| --- | --- | --- |

| 结构树与构件定位 | 可保留 | 已具备基础能力，优化搜索、过滤、绑定状态标识。 |

| 主辅机参数与台账展示 | 需修正文档 | 代码已有设备台账、结构参数、运行参数展示，后续应审查真实数据来源和字段完整性。 |
| 模型资产选择与缩略图 | 可保留 | 已具备基础版本，继续优化筛选、分页、状态标签、懒加载。 |

| 扫描建模定期更新 | 不应归为前端单项需求 | 属于模型生产和发布流程，需要模型侧、项目管理、后台、前端共同定义。 |
| 4D 进度模拟 | 后续阶段 | 需要 WBS、计划任务和构件绑定，不建议当前阶段进入 P0/P1。 |

| AI 助理 | 暂不纳入当前需求 | 需要文档、模型、KKS、台账统一索引后再评估。 |

## 三、需求问题清单

### 1. “全厂级”范围没有可验收边界

问题：

- 文档中出现全厂漫游、办公楼、厂房、道路、绿植、路灯、主辅机、管道、阀门、仪表等描述。
- 但缺少全厂资产清单、模型资源目录、区域清单、建模完成度说明。

影响：

- 前端即使具备三维平台能力，也无法证明满足“全厂级”。
- 验收时容易把“平台能力完成”误判为“全厂资产交付完成”。

审查建议：

- 必须先输出全厂资产覆盖矩阵。
- 没有资产矩阵时，需求口径应改为“局部区域示范场景”或“具备全厂级扩展能力”。

最低验收字段：

| 字段 | 说明 |
| --- | --- |
| 资产类型 | 办公楼、厂房、道路、主机、辅机、管道、阀门、仪表等。 |
| 资产名称 | 具体资产名称。 |
| 所属区域 | 厂区、楼栋、系统或专业。 |
| 是否建模 | 是/否。 |
| 是否入场景 | 是/否。 |
| 是否可定位 | 是/否。 |
| 是否可选中 | 是/否。 |
| 是否绑定 KKS/设备/文档 | 是/否。 |
| 数据来源 | 模型、KKS 台账、设备台账、实时接口、人工录入。 |

### 2. “实时高清渲染”缺少性能指标

问题：

- “实时高清”不是可执行需求。
- 未定义模型规模、设备配置、分辨率、帧率、加载时间、内存上限。

影响：

- 大模型加载慢、帧率低时难以判断是缺陷还是资源规模超出边界。

审查建议：

- 将需求改写为可量化指标。
- 性能指标必须绑定模型规模和运行环境。

建议验收口径：

| 指标 | 建议定义 |
| --- | --- |
| 首屏可交互时间 | 指定模型规模下，进入页面后 N 秒内可旋转/缩放。 |
| 稳态帧率 | 指定浏览器和设备下，普通浏览保持最低 FPS。 |
| 加载失败处理 | 模型、贴图、缩略图失败时必须展示原因和重试入口。 |
| 质量档 | 高/中/低/平面渲染必须有明确降级策略。 |
| 资源释放 | 切换模型或退出页面后释放 geometry、material、texture、ObjectURL。 |

### 3. 参数台账展示已实现基础能力，但真实数据口径未闭环

问题：

- 代码中 `ObjectInfoPanel.vue` 已有设备台账、结构参数、运行参数、绑定文件、测点关联。
- 旧文档仍把“未展示设备台账/参数”列为核心缺口。
- 当前真正缺口应调整为“真实数据来源、字段完整性、数据更新时间和异常状态”。

审查建议：

- 更新需求文档，不再表述为“新增设备台账展示”。
- 改为“参数台账数据闭环与字段标准化”。

必须澄清：

- 哪些设备属于主机和重要辅机。
- 每类设备必须展示哪些台账字段。
- 哪些字段来自 KKS 台账，哪些来自设备台账，哪些来自实时接口。
- 参数刷新频率、更新时间、离线/异常状态如何展示。
- 演示数据是否允许进入验收截图。

### 4. 导航图已存在基础能力，但仍缺少验收用配置

问题：

- 当前代码已有 `NavigationMapPanel.vue`，并支持热点、系统/书签/设备定位。
- 旧文档仍写“缺少图形化导航图”。
- 真实需求应从“有没有导航图”改为“导航图是否可验收”。

审查建议：

- 更新需求为“导航图增强和验收配置”。

建议验收口径：

- 有导航图底图或快照。
- 有区域/系统/书签/设备热点。
- 点击热点可定位到三维视角。
- 当前选中设备可在导航图上高亮。
- 导航图失败时有刷新或重新生成入口。
- 平面渲染模式下有明确交互限制提示。

### 5. 模型加载鉴权属于 P0 稳定性需求

问题：

- 之前已出现 `/visualization/3d-fullscreen` 模型下载匿名请求失败问题。
- 三维页模型、缩略图、贴图、LOD 资源、文档预览都可能涉及鉴权。

审查建议：

- 将“模型资源鉴权加载”列为 P0，而不是普通缺陷。

验收标准：

- GLB/GLTF/IFC/OBJ 下载请求携带鉴权信息。
- 缩略图请求携带鉴权信息。
- 贴图、bin、LOD 子资源不绕过鉴权。
- token 过期时有明确提示，不静默失败。
- 加载失败展示 URL、状态、错误原因和重试入口。

### 6. 扫描建模定期更新不是单一前端需求

问题：

- 该需求包含扫描数据、模型清洗、贴图处理、LOD 生成、压缩减面、资源发布、版本回溯。
- 前端只能消费发布产物，不能单独完成扫描建模闭环。

审查建议：

- 将该需求拆成“模型生产流程需求”和“前端消费需求”。

模型生产流程需求：

- 扫描数据输入规范。
- 模型清洗与减面规范。
- 贴图和材质规范。
- LOD/压缩产物规范。
- 发布包目录规范。
- 模型版本台账。

前端消费需求：

- 支持按版本加载模型。
- 支持读取 `scene-manifest`。
- 支持 LOD、质量档、资源分组。
- 支持模型更新时间、版本号、发布说明展示。

### 7. 4D/5D、BIM+GIS、AI 当前不应混入验收主线

问题：

- 这些能力来自广联达完整产品矩阵，但不是当前三维底座自然完成项。
- 如果直接纳入当前验收，会显著扩大范围。

审查建议：

- 当前阶段只保留接口和架构扩展性。
- 不纳入当前版本验收。

进入条件：

| 能力 | 前置条件 |
| --- | --- |
| 4D 进度模拟 | WBS、计划任务、构件或资源分组绑定关系。 |
| 5D 成本物资 | 成本科目、物资台账、工程量或设备清单。 |
| BIM+GIS | GIS 底图、坐标系、厂区空间数据、模型定位基准。 |
| AI 助理 | 文档、模型、KKS、台账统一索引和权限边界。 |

## 四、建议重排需求优先级

### P0：验收收口需求

| 需求 | 验收方式 | 责任边界 |
| --- | --- | --- |
| 模型加载鉴权稳定 | 使用受限模型在全屏页正常加载，失败时有提示和重试 | 前端/后台 |
| 模型资产选择体验 | 可搜索、可看缩略图、可多选、可识别已加载模型 | 前端/后台 |
| 构件属性与业务数据展示 | 选中构件后展示 KKS、台账、参数、文档、测点 | 前端/业务/后台 |
| 数据来源和模拟数据标识 | 每类字段明确来源和更新时间 | 业务/后台/前端 |
| 全厂资产覆盖矩阵 | 用清单证明已建模、已入场景、可定位、可绑定 | 模型侧/业务 |
| 场景保存与恢复 | 模型、点位、摄像头、视角、图层、测量、剖切可恢复 | 前端/后台 |
| 验收矩阵更新 | 需求、证据、测试数据、截图、负责人一一对应 | 产品/测试/项目 |

### P1：体验与协同增强

| 需求 | 验收方式 | 责任边界 |
| --- | --- | --- |
| 导航图增强 | 热点可定位、设备可高亮、支持刷新和失败提示 | 前端/模型侧 |
| 构件绑定操作 | 可从构件发起绑定 KKS、文档、测点、摄像头 | 前端/后台 |
| 文档预览联动 | 构件绑定文档可直接预览，文档可回到构件 | 前端/后台 |
| 模型批注/问题闭环 | 保存视点、构件、截图、责任人、状态 | 前端/后台/业务 |
| 告警定位与着色 | 告警设备可在模型中高亮并定位 | 前端/后台/实时数据 |
| 模型版本元数据 | 可查看模型版本、上传人、发布时间、变更说明 | 前端/后台 |

### P2：平台扩展能力

| 需求 | 进入条件 |
| --- | --- |
| 4D 进度模拟 | 有 WBS 和构件绑定。 |
| 5D 成本物资 | 有成本、物资、合同或工程量数据。 |
| BIM+GIS | 有 GIS 底图、坐标系和厂区空间数据。 |
| 扫描建模更新闭环 | 模型侧能提供标准发布包和版本台账。 |
| AI 助理 | 数据统一索引、权限、问答边界明确。 |

## 八、最终建议

当前需求应先做“需求收口”，再继续开发。建议按以下顺序推进：

1. 先更新旧文档中已过时的判断，尤其是导航图和设备台账展示。
2. 补全资产覆盖矩阵，明确当前是局部示范还是全厂级交付。
3. 将真实数据、演示数据、模拟数据在页面和验收材料中区分。
4. 将模型加载鉴权和资源加载稳定性列为 P0。
5. 将模型批注、文档联动、版本追溯列为 P1。
6. 将 4D/5D、BIM+GIS、AI、扫描建模闭环列为 P2 或专项需求，不进入当前版本验收主线。

一句话判断：

`当前三维 BIM 需求可以继续推进，但必须从“大而全的全厂级 BIM 平台”收敛为“可验证、可交付、可扩展的三维数字交付底座”。`
