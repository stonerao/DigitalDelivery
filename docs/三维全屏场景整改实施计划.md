# 三维全屏场景整改实施计划

更新时间：2026-04-24

适用范围：`/visualization/3d-fullscreen` 三维全屏场景能力补齐与全厂级需求整改推进。

## 一、目标

基于当前核查结果，本计划用于推动三维全屏场景从“具备三维巡视平台能力和局部场景展示”逐步补齐到“满足全厂级交付要求”。

本次整改目标聚焦以下四类问题：

1. fullscreen 设备参数与台账展示不足
2. 缺少图形化导航图
3. 缺少全厂资产覆盖与验收证据
4. 缺少扫描建模更新闭环

## 二、整改原则

### 1. 先补验收短板，再补长期能力

优先处理会直接影响需求验收结论的问题：

- fullscreen 参数台账展示
- 导航图
- 全厂资产覆盖清单

离线资产处理流水线和扫描更新闭环属于中长期能力建设，应纳入后续阶段，但不应阻塞前两项的交付收口。

### 2. 先形成证据链，再扩展系统能力

当前部分问题不是“完全没有实现”，而是“没有证据证明满足”。因此整改时应同步产出：

- 功能实现
- 资产清单
- 验收矩阵
- 使用说明或演示路径

### 3. 平台能力与场景内容分开推进

- 平台能力：由前端/三维运行时负责
- 场景内容：由模型、工程包、资产配置、业务数据侧负责

若两者混在一起推进，容易导致“功能完成，但无法验收”。

## 三、整改分级

### P0：本阶段必须完成

1. fullscreen 设备参数与台账展示补齐
2. 输出全厂资产覆盖清单
3. 明确当前项目交付范围（局部区域 or 全厂级）

### P1：本阶段建议完成

1. 增加图形化导航图/小地图
2. 打通主机和重要辅机真实参数展示
3. 建立导航图与区域/系统/书签联动

### P2：后续建设项

1. 建设离线资产处理流水线
2. 建设扫描建模更新闭环
3. 完善版本管理、发布包与资源目录规范

## 四、实施任务分解

### 任务 1：补 fullscreen 设备参数与台账展示

优先级：`P0`

目标：

在 fullscreen 选中设备后，展示完整的业务资料卡，达到需求文档中的“显示主机和重要辅机设备参数，包括台账信息”口径。

整改内容：

1. 在 fullscreen 对象信息面板中增加以下信息块：
   - 台账基础信息
   - 结构参数
   - 运行参数
   - 更新时间/数据来源
2. 复用普通页已有的设备资料结构，避免维护两套展示逻辑。
3. 对演示数据和真实数据增加显式标识。

建议产出物：

- fullscreen 设备信息面板升级代码
- 字段映射说明
- 演示截图

验收标准：

- 可在 fullscreen 中看到设备台账号、厂家、型号、额定参数、结构参数、运行参数
- 重要辅机与主机至少各有一类样例可完整展示
- 字段为空时有缺省态，不出现空白错误或错位

前置依赖：

- 业务侧确认台账字段和运行参数字段标准
- 数据源确认是否为演示数据或真实数据

风险：

- 若后台数据结构不统一，前端展示完成后仍可能出现缺字段

### 任务 2：输出全厂资产覆盖清单

优先级：`P0`

目标：

将需求中的“办公楼、厂房、道路、绿植、路灯、宣传栏、雕像、主辅机、管道、阀门、仪表”等资产形成可验收清单。

整改内容：

1. 建立统一资产覆盖表。
2. 对每类资产标注：
   - 是否已建模
   - 是否已入场景
   - 是否可定位
   - 是否已绑定业务数据
3. 建立“未完成项”清单。

建议产出物：

- 全厂资产覆盖 Excel 或 Markdown 台账
- 资产与场景节点映射表
- 资产缺失列表

验收标准：

- 能逐项回答需求中每类资产是否已纳入场景
- 可明确说明缺失范围和补齐计划

前置依赖：

- 模型侧提供全厂资产清单
- 工程包侧提供当前场景资源目录

风险：

- 若当前项目实际只建了局部区域，资产清单会直接暴露阶段边界，需要同步调整对外口径

### 任务 3：明确当前交付范围

优先级：`P0`

目标：

避免“局部区域示范场景”按“全厂级正式交付”口径被验收。

整改内容：

1. 明确当前项目定位：
   - 局部区域示范
   - 多区域组合展示
   - 全厂级交付
2. 将交付边界写入项目说明、汇报材料与验收文档。
3. 对未纳入范围的内容形成例外说明。

建议产出物：

- 项目范围说明
- 阶段性边界说明
- 对外汇报统一口径

验收标准：

- 项目组、业务方、验收方对当前交付范围达成一致
- 验收会议中不再出现“局部场景按全厂口径验收”的歧义

前置依赖：

- 业务方确认本期范围
- 项目经理确认阶段目标

风险：

- 若前期对外口径已按全厂宣传，调整范围时可能产生预期偏差

### 任务 4：增加图形化导航图

优先级：`P1`

目标：

实现需求中的“可以瞬间定位的导航图”。

整改内容：

1. 增加厂区导航图或区域小地图组件。
2. 支持点击区域定位至场景对应位置或书签视角。
3. 支持当前区域高亮、当前位置同步。
4. 导航图与系统节点、书签、区域视角建立映射。

建议产出物：

- 导航图组件
- 区域映射配置
- 演示路径文档

验收标准：

- 用户可通过导航图一键跳转到指定区域
- 导航图与三维视角存在联动
- 至少覆盖锅炉区域、控制室、冷却塔等已有区域

前置依赖：

- 需确认厂区总图或区域平面图来源
- 需定义区域与场景坐标/视角映射关系

风险：

- 若没有统一厂区总图，导航图很容易变成临时拼图，后续维护成本高

### 任务 5：打通真实参数展示

优先级：`P1`

目标：

将当前示例/演示参数逐步替换为真实业务参数或明确标识为模拟数据。

整改内容：

1. 梳理主机、重要辅机的参数展示字段标准。
2. 统一实时数据接入标识、更新时间、质量状态。
3. 将重要设备的真实参数优先接入 fullscreen。

建议产出物：

- 参数字段白名单
- 数据接入映射表
- 真实/模拟数据标识规则

验收标准：

- 至少核心主机和关键辅机的参数可在 fullscreen 中完整展示
- 用户可区分真实数据与模拟数据

前置依赖：

- 后台提供参数接口和字段说明
- 业务方确认关键设备清单

风险：

- 若参数来源分散，统一展示口径需要较多协调

### 任务 6：建设离线资产处理流水线

优先级：`P2`

目标：

建立从模型原始产物到运行时资源产物的离线处理流程。

整改内容：

1. 贴图整理与标准化
2. 材质整理
3. 模型分组
4. LOD 生成
5. 压缩与减面
6. 发布包与 `scene-manifest` 产物规范

建议产出物：

- 流水线脚本或处理工具
- 发布目录规范
- 产物校验规则

验收标准：

- 可以稳定输出运行时可消费的工程包
- 质量档和 LOD 不再依赖纯人工拼装

前置依赖：

- 模型侧提供原始产物规范
- 工程包侧提供发布标准

风险：

- 离线流水线涉及模型、前端、发布三侧，推进周期较长

### 任务 7：建设扫描建模更新闭环

优先级：`P2`

目标：

实现“现场扫描 -> 模型更新 -> 工程包发布 -> 场景更新”的闭环。

整改内容：

1. 定义扫描数据输入标准。
2. 建立模型更新版本记录。
3. 形成工程包发布和回溯机制。
4. 让运行时可按版本切换或回看。

建议产出物：

- 扫描建模更新流程图
- 模型版本台账
- 变更记录模板

验收标准：

- 可追踪某次扫描对应的模型版本和发布时间
- 场景更新具备可回溯性

前置依赖：

- 业务方与模型侧确认扫描更新周期
- 发布侧提供版本管理规范

风险：

- 若扫描与建模周期不稳定，该闭环难以标准化

## 五、建议实施顺序

### 第一阶段：验收收口

目标：先解决当前最直接影响需求评审的问题。

建议顺序：

1. 明确交付范围
2. 补 fullscreen 参数台账
3. 输出全厂资产覆盖清单

阶段成果：

- 可以对外给出清晰、可落地的验收口径
- 可以减少“功能有了但无法证明满足”的问题

### 第二阶段：体验补齐

目标：补齐需求中体验层面的关键缺口。

建议顺序：

1. 增加导航图
2. 打通导航图与区域视角联动
3. 打通真实参数展示

阶段成果：

- 三维全屏场景更接近正式业务使用体验

### 第三阶段：能力建设

目标：建设长期可持续更新的资产和场景体系。

建议顺序：

1. 离线资产处理流水线
2. 扫描建模更新闭环
3. 发布和版本回溯规范

阶段成果：

- 形成模型持续更新能力
- 降低后续前端运行时补丁式优化压力

## 六、角色建议

| 事项 | 建议责任角色 | 协同角色 |
| --- | --- | --- |
| fullscreen 参数台账展示 | 前端 / 三维运行时 | 后台、业务 |
| 全厂资产覆盖清单 | 模型侧 / 业务侧 | 项目经理、前端 |
| 范围确认 | 项目经理 / 业务方 | 全体 |
| 导航图实现 | 前端 / 产品 | 模型侧、业务 |
| 真实参数接入 | 后台 / 业务 | 前端 |
| 离线资产流水线 | 模型侧 / 工程化 | 前端、发布 |
| 扫描建模更新闭环 | 模型侧 / 项目管理 | 发布、业务 |

## 七、建议管理方式

### 1. 周推进表

建议每周跟踪以下内容：

- 本周完成项
- 本周阻塞项
- 下周计划
- 风险变化

### 2. 统一文档入口

建议将以下文档作为同一组交付材料管理：

- 需求符合性核查清单
- 问题清单与整改建议（简版）
- 验收矩阵
- 整改实施计划

### 3. 用验收矩阵反推开发范围

开发任务不应只按“功能点”排，还应按“验收能否闭环”排。  
即：每一项整改任务都必须能在验收矩阵里找到对应完成项。

## 八、建议对外说明

建议在阶段汇报中使用以下表达：

`当前阶段已完成三维巡视平台核心能力建设，包括模型浏览、漫游、定位、点位、摄像头、测量和业务绑定。后续整改将重点补齐 fullscreen 参数台账展示、导航图、全厂资产覆盖证据以及扫描建模更新闭环，以满足全厂级场景交付要求。`
