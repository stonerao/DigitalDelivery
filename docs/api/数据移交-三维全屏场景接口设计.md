# 数据移交-三维全屏场景接口设计

目标页面：`/visualization/3d-fullscreen`

对应前端实现：

- `src/views/visualization/viewer3d/fullscreen.vue`
- `src/views/visualization/viewer3d/services/projectPackageService.js`
- `src/views/visualization/viewer3d/services/sceneAnchorService.js`
- `src/views/visualization/viewer3d/services/sceneSchemeService.js`

## 1. 设计目标

当前页面已经具备以下能力，但后端接口尚未落地：

- 模型加载与多 LOD/质量切换
- 场景方案保存/应用
- 点位与摄像头点位管理
- 测量结果保存
- 书签保存
- 剖切状态与剖切预设
- 设备定位、系统定位、分层显隐
- 构件属性绑定与编辑
- 实时点位接入配置
- 后端桥接配置
- 脚本触发器与动画动作
- 资源分组与场景清单

因此后端不建议只补一个“保存按钮接口”，而应提供一套围绕“3D 场景工程包”的接口。

## 2. 总体方案

建议以 `viewerProject` 作为核心资源，一份模型对应一个三维场景工程包。

工程包建议拆成四个一级域：

- `scene`：场景配置与可视化状态
- `scripts`：脚本、触发器、动画
- `assets`：场景清单、资源分组、LOD
- `integrations`：实时数据接入与后端桥接

页面保存时推荐两种方式并存：

- 方式 A：整包读取，分块保存
- 方式 B：关键高频资源单独增删改查

这样既能满足当前前端的快速落地，也能支撑后续多人协作、权限、版本和审计。

## 3. 核心数据模型

## 3.1 ViewerProject

```json
{
  "id": "vp_10001",
  "projectCode": "viewer-3d-demo",
  "projectName": "3D全屏场景工程",
  "modelId": "model_001",
  "modelName": "汽机房模型",
  "modelUrl": "/model/test.glb",
  "version": 12,
  "updatedAt": "2026-04-07T10:00:00Z",
  "updatedBy": "admin"
}
```

说明：

- `id` 为后端工程主键
- `modelId` 对应模型管理页面的模型记录
- `version` 用于乐观锁，避免多人覆盖

## 4. 接口设计

## 4.1 页面初始化

- 工程列表：`GET /api/viewer/projects`
- 工程详情：`GET /api/viewer/projects/{projectId}`
- 按模型获取工程：`GET /api/viewer/projects/by-model?modelId={modelId}`
- 场景工程包：`GET /api/viewer/projects/{projectId}/package`
- 获取运行态快照：`GET /api/viewer/projects/{projectId}/runtime-snapshot`

页面行为：

- 进入 `/visualization/3d-fullscreen?id=xxx` 时先按 `modelId` 获取工程
- 若不存在工程，可调用“创建工程”
- 获取工程包后一次性恢复书签、点位、测量、剖切、脚本、集成配置

## 4.2 工程创建与保存

- 创建工程：`POST /api/viewer/projects`
- 更新工程基础信息：`PUT /api/viewer/projects/{projectId}`
- 保存整个工程包：`PUT /api/viewer/projects/{projectId}/package`
- 分块保存工程包：`PATCH /api/viewer/projects/{projectId}/package`

创建工程请求示例：

```json
{
  "projectCode": "viewer-model-001",
  "projectName": "汽机房全屏场景",
  "modelId": "model_001",
  "modelName": "汽机房模型",
  "modelUrl": "/model/test.glb"
}
```

分块保存请求示例：

```json
{
  "version": 12,
  "scene": {
    "bookmarks": [],
    "deviceBindings": [],
    "anchors": [],
    "cameras": [],
    "measurements": [],
    "schemes": [],
    "clipping": {},
    "clippingPresets": [],
    "viewerState": {}
  }
}
```

说明：

- 推荐 `PUT /package` 用于“保存全部”
- 推荐 `PATCH /package` 用于前端局部保存，减少并发冲突
- 请求体必须带 `version`
- 版本不一致返回 `409`

并发控制建议：

- 所有子资源接口也应支持 `version`
- 推荐两种实现方式二选一：
- 方式 A：请求体带 `version`
- 方式 B：请求头带 `If-Match: "{version}"`
- 后端保存成功后返回最新 `version`
- 若版本冲突，返回 `409`，同时返回 `currentVersion`

## 4.3 场景方案

- 查询场景方案列表：`GET /api/viewer/projects/{projectId}/scene-schemes`
- 新增场景方案：`POST /api/viewer/projects/{projectId}/scene-schemes`
- 修改场景方案：`PUT /api/viewer/projects/{projectId}/scene-schemes/{schemeId}`
- 删除场景方案：`DELETE /api/viewer/projects/{projectId}/scene-schemes/{schemeId}`

说明：

- 如果要快速上线，也可以只保留 `PATCH /package`
- 但从后续复用角度，`scene-schemes` 建议作为独立资源
- 新增/修改/删除时同样建议携带 `version`，避免用户 A 保存方案时覆盖用户 B 刚更新的工程状态

## 4.4 书签

- 查询书签：`GET /api/viewer/projects/{projectId}/bookmarks`
- 新增书签：`POST /api/viewer/projects/{projectId}/bookmarks`
- 修改书签：`PUT /api/viewer/projects/{projectId}/bookmarks/{bookmarkId}`
- 删除书签：`DELETE /api/viewer/projects/{projectId}/bookmarks/{bookmarkId}`

说明：

- 书签截图缩略图可由后端接收 base64 或文件上传生成
- 若第一期不做缩略图，可先只存相机参数
- 修改/删除时建议校验 `version`

## 4.5 点位与摄像头点位

- 查询普通点位：`GET /api/viewer/projects/{projectId}/anchors`
- 新增普通点位：`POST /api/viewer/projects/{projectId}/anchors`
- 修改普通点位：`PUT /api/viewer/projects/{projectId}/anchors/{anchorId}`
- 删除普通点位：`DELETE /api/viewer/projects/{projectId}/anchors/{anchorId}`
- 查询摄像头点位：`GET /api/viewer/projects/{projectId}/camera-anchors`
- 新增摄像头点位：`POST /api/viewer/projects/{projectId}/camera-anchors`
- 修改摄像头点位：`PUT /api/viewer/projects/{projectId}/camera-anchors/{anchorId}`
- 删除摄像头点位：`DELETE /api/viewer/projects/{projectId}/camera-anchors/{anchorId}`

说明：

- 若后端更偏向统一建模，也可以统一成 `/anchors`，通过 `type=camera-point` 区分
- 当前前端实现中普通点位和摄像头点位字段不同，分接口更直接
- 点位修改频率高，建议接口返回最新 `version`

## 4.6 测量结果

- 查询测量结果：`GET /api/viewer/projects/{projectId}/measurements`
- 新增测量结果：`POST /api/viewer/projects/{projectId}/measurements`
- 批量覆盖测量结果：`PUT /api/viewer/projects/{projectId}/measurements`
- 修改显隐：`PATCH /api/viewer/projects/{projectId}/measurements/{measurementId}`
- 删除测量结果：`DELETE /api/viewer/projects/{projectId}/measurements/{measurementId}`
- 清空测量结果：`DELETE /api/viewer/projects/{projectId}/measurements`

说明：

- 当前前端在 `measure-change` 时拿到全量 `records`
- 因此前端第一期最省事的方式是使用 `PUT /measurements` 全量覆盖
- 如果后续支持多人同时编辑测量结果，应引入行级 `updatedAt/updatedBy` 和包级 `version`

## 4.7 剖切状态与剖切预设

- 获取剖切状态：`GET /api/viewer/projects/{projectId}/clipping/state`
- 保存剖切状态：`PUT /api/viewer/projects/{projectId}/clipping/state`
- 查询剖切预设：`GET /api/viewer/projects/{projectId}/clipping/presets`
- 新增剖切预设：`POST /api/viewer/projects/{projectId}/clipping/presets`
- 删除剖切预设：`DELETE /api/viewer/projects/{projectId}/clipping/presets/{presetId}`

说明：

- 剖切动画是否保存，建议单独增加字段 `animation`
- 若第一期无需恢复动画播放中状态，只保存静态 `clippingState`
- 剖切状态自动保存频率高，强烈建议做版本控制或最后写入时间校验

## 4.8 构件业务绑定

- 查询构件绑定：`GET /api/viewer/projects/{projectId}/device-bindings`
- 批量保存构件绑定：`PUT /api/viewer/projects/{projectId}/device-bindings`
- 单条新增/修改：`PUT /api/viewer/projects/{projectId}/device-bindings/{bindingId}`
- 按构件 UUID 查询：`GET /api/viewer/projects/{projectId}/device-bindings/by-object?objectUuid={uuid}`
- 删除绑定：`DELETE /api/viewer/projects/{projectId}/device-bindings/{bindingId}`

说明：

- 这组接口是本次最关键的遗漏之一
- 页面右键“绑定属性/修改属性/清空属性”最终都应落到这里
- 批量保存建议走覆盖式 `PUT`，单条编辑建议走 `PUT /{bindingId}`，两种都要带 `version`

## 4.9 实时接入与后端桥接配置

- 获取实时接入配置：`GET /api/viewer/projects/{projectId}/integrations/realtime`
- 保存实时接入配置：`PUT /api/viewer/projects/{projectId}/integrations/realtime`
- 获取后端桥接配置：`GET /api/viewer/projects/{projectId}/integrations/backend-bridge`
- 保存后端桥接配置：`PUT /api/viewer/projects/{projectId}/integrations/backend-bridge`

## 4.10 ViewerState 与界面偏好

- 获取界面状态：`GET /api/viewer/projects/{projectId}/viewer-state`
- 保存界面状态：`PUT /api/viewer/projects/{projectId}/viewer-state`
- 局部更新界面状态：`PATCH /api/viewer/projects/{projectId}/viewer-state`

说明：

- 如果后端希望极简，也可以继续合并在 `scene.viewerState`
- 但从接口语义上拆出 `viewer-state` 更清楚，尤其适合自动保存
- `viewer-state` 变更频繁，也应支持 `version`

## 4.11 脚本与触发器

- 获取脚本定义：`GET /api/viewer/projects/{projectId}/scripts`
- 全量保存脚本定义：`PUT /api/viewer/projects/{projectId}/scripts`
- 获取动画列表：`GET /api/viewer/projects/{projectId}/scripts/animations`
- 获取触发器列表：`GET /api/viewer/projects/{projectId}/scripts/triggers`

说明：

- 当前页面使用的是整包存储思路
- 后端一期可以只实现 `PUT /scripts`

## 4.12 场景清单与资源分组

- 获取场景清单：`GET /api/viewer/projects/{projectId}/scene-manifest`
- 保存场景清单：`PUT /api/viewer/projects/{projectId}/scene-manifest`
- 获取资源分组：`GET /api/viewer/projects/{projectId}/asset-groups`
- 保存资源分组：`PUT /api/viewer/projects/{projectId}/asset-groups`

说明：

- `groups` 除了 `uuids` 外，建议保留 `visible`、`modelUrl`、`sort`、`ext`
- 这样能覆盖当前页面的资源组显隐，也能给后续“按组切换模型资源”留位置

## 4.13 运行态数据接口

这些接口不属于“场景配置保存”，但当前页面运行时会依赖。

- 实时点位订阅令牌：`POST /api/viewer/runtime/realtime/session`

## 6.3 自动保存

推荐自动保存以下高频数据：

## 8. 异常码建议

## 10. 结论

`/visualization/3d-fullscreen` 的保存能力本质上不是单一“节点/标记/点位保存”，而是一个完整的三维场景工程保存。

后端建议优先实现：

- 工程主表 `viewer_projects`
- 工程包接口 `/package`
- 构件绑定接口 `/device-bindings`

只要这三部分先落地，前端当前页面的大部分功能就能真正从本地存储切换到后端持久化。
