# BIM三维平台技术分册

日期：2026-03-25

本次状态更新：2026-04-02（按当前仓库实现对照标注）

状态标记说明：

- `已开发`：仓库中已有可用实现，且已接入页面或运行链路
- `部分开发`：仓库中已有骨架或局部实现，但未完全达到本分册目标
- `未开发`：仓库中尚无对应实现，或仅停留在文档/字段预留层面

## 1. 文档目标

本文档用于明确 BIM 三维平台的总体技术方案与分阶段开发计划，作为后续功能设计、任务拆分、接口定义、联调验收的统一基线。

当前范围聚焦以下六类能力，并按当前项目状态标注如下：

1. 场景点位编辑与展示 `部分开发`
2. 摄像头点位编辑、展示与视频接入 `部分开发`
3. 多类型测量能力扩展 `已开发`
4. 模型后处理与图形优化 `部分开发`
5. 三维引擎平台数据交互接口 `部分开发`
6. 工程应用配置工具 `部分开发`

## 2. 现状与问题

当前三维页面仍以 `src/views/visualization/viewer3d/fullscreen.vue` 为核心，但运行时能力已经开始从“大页组件”向 `toolbars`、`panels`、`services`、`store`、`runtime` 模块拆分。

底层三维能力主要由 `src/3d/components/ThreeModelViewer.vue` 提供，当前已确认具备：

- 模型加载：GLTF / GLB / IFC / OBJ `已开发`
- 相机控制：旋转、平移、缩放、正交/透视、预设视角、第一人称漫游 `已开发`
- 场景操作：拾取、高亮、聚焦、隔离、按 UUID 显隐 `已开发`
- 工具能力：剖切、剖切动画、两点/折线/夹角/面积测量、截图、书签 `已开发`
- 性能能力：BVH、动态像素比、材质降级、纹理预算控制 `已开发`
- 点位/摄像头渲染与点击拾取 `已开发`

当前项目内已经存在的上层能力：

- 运行时页面拆分为顶部工具栏、底部工具栏、对象信息、结构树、点位、测量、联动、资源分组、剖切等面板 `已开发`
- 运行时状态与工程配置状态分别落在 `viewer3dRuntime`、`viewer3dProject` store `已开发`
- 场景点位、摄像头点位、测量结果、场景方案、剖切配置写入工程包 `已开发`
- `RealtimeDataAdapter`、`ScriptEngine`、`BackendBridge` 已有实现并接入运行时 `部分开发`
- 独立的 `configStudio` 页面已存在，支持工程包导入导出、脚本编辑、集成配置、清单编辑、校验 `部分开发`

现阶段仍然存在的主要问题：

- `fullscreen.vue` 体量仍然偏大，运行时编排与具体实现尚未完全分离 `部分开发`
- 点位与摄像头已基本统一到锚点模型，但测量、方案、资产分组仍是并列模型，尚未形成更完整的统一场景对象模型 `部分开发`
- 工程配置端已存在，但“工程目录编辑”“点位/摄像头在配置端独立维护”尚未完成 `部分开发`
- 资产清单与质量/LOD 切换已存在，但离线后处理流水线尚未落地 `部分开发`
- 数据交互层当前以 `mock/http` 为主，WebSocket、MQTT、长连接任务同步等能力尚未实现 `部分开发`

## 3. 建设目标

总体目标是将现有 BIM 三维页面升级为“可配置、可扩展、可交付”的三维平台，而不是单一场景浏览页。

建设结果应满足：

- 运行时场景支持工程级点位、视频、测量、实时数据、动画联动 `部分开发`
- 工程配置能力独立，支持项目结构、点位、脚本、后台程序配置 `部分开发`
- 模型资源经过离线处理后再投产，保证加载性能与画质可控 `未开发`
- 平台具备统一的数据交互接口和脚本执行机制 `部分开发`
- 页面层、引擎层、配置层、资产层职责清晰 `部分开发`

## 9. 当前优先级建议

基于当前已开发情况，后续优先顺序建议调整为：

1. 完成资产离线处理流水线
2. 完成配置端的工程目录、点位、摄像头编辑能力
3. 继续拆分 `fullscreen.vue`，补齐 `anchors` / `measurements` / `composables` 目录
4. 扩展实时与视频协议（WebSocket、MQTT、HLS、WebRTC）
5. 强化脚本动作集与后台长连接能力

原因：

- 当前最明显的短板已经不是基础浏览，而是“可交付”和“可配置”
- 运行时核心能力已具备，后续应优先补齐工程化闭环

## 10. 风险与控制点

主要风险：

- `fullscreen.vue` 持续膨胀导致后续维护成本升高
- 点位、摄像头、测量、脚本联动继续叠加后，状态管理复杂度上升
- 实时/后台适配层目前协议覆盖有限，后续接真实后端时可能频繁调整
- 没有离线资产流水线时，运行时性能问题仍会被迫在前端兜底
- 配置端与运行端若长期并存编辑入口，容易造成工程包来源不一致

控制建议：

- 继续以工程包为唯一配置源，运行时只消费
- 点位、摄像头、测量、方案统一围绕工程包和事件总线扩展
