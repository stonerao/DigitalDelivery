# BIM三维运行时架构拆分设计

日期：2026-03-25

## 1. 文档目标

本文档用于细化 BIM 三维平台运行时部分的模块拆分方案，明确现有页面如何从“大页面集中编排”演进为“内核、状态、面板、工具、业务联动”分层结构。

本文档对应 [技术分册.md](e:/项目/数字交付/DigitalDelivery/docs/技术分册.md#L1) 中的“阶段一：运行时架构拆分”。

## 2. 现有问题

当前 [fullscreen.vue](e:/项目/数字交付/DigitalDelivery/src/views/visualization/viewer3d/fullscreen.vue#L1) 直接承载以下职责：

- 页面布局与面板切换
- 三维 Viewer 控制
- 结构树、导航树、业务设备树
- 显示模式、剖切、漫游、截图、书签
- 测点联动与对象属性展示
- 场景方案持久化
- 构件业务属性编辑

问题在于：

- 页面状态过多，后续点位、摄像头、动画、实时数据接入后会继续膨胀
- `viewerRef` 暴露方法分散调用，后续容易形成难以维护的命令耦合
- 运行态业务状态、场景状态、工具状态混在同一文件中
- 面板与工具逻辑没有形成稳定边界，不利于增量扩展

## 3. 运行时目标结构

建议将运行时拆分为五层。

### 3.1 ViewerAdapter 层

定位：页面与底层 `ThreeModelViewer` 之间的适配层。

职责：

- 对外提供统一命令接口
- 封装 `viewerRef` 调用
- 统一处理模型加载、选择、高亮、聚焦、隔离、显隐、剖切、测量、截图等底层能力
- 统一转发底层事件到运行时 store

目标效果：

- 页面和面板不直接依赖 `viewerRef.value?.xxx`
- 后续底层 Viewer 替换或增强时，业务层改动最小

### 3.2 Runtime Store 层

定位：运行时状态层。

建议拆分为两个 store。

#### viewer3dRuntime

负责运行时即时状态：

- 当前工具模式
- 当前选中对象
- 当前显示模式
- 漫游状态
- 剖切状态
- 点位显隐
- 视频窗口状态
- 当前激活面板和侧栏

#### viewer3dProject

负责工程级运行数据：

- 场景方案
- 点位数据
- 摄像头数据
- 测量结果
- 运行时加载的工程包
- 业务绑定关系缓存

### 3.3 Runtime Service 层

定位：运行时功能服务层。

建议独立模块：

- `selectionService`
- `visibilityService`
- `clippingService`
- `measurementService`
- `anchorService`
- `cameraService`
- `schemeService`
- `realtimeService`
- `scriptRuntimeService`

职责：

- 组织复杂功能逻辑
- 调用 `ViewerAdapter`
- 更新运行时 store
- 对面板和工具栏提供统一服务接口

### 3.4 Runtime UI 层

定位：运行时 UI 组件层。

建议拆成：

- 顶栏
- 底部工具栏
- 左侧对象属性与视频面板
- 右侧业务与结构侧栏
- 剖切控制面板
- 点位与摄像头编辑面板
- 测量结果面板

UI 组件只负责：

- 展示状态
- 发出用户操作
- 调用 service 或 dispatch store action

不负责：

- 直接操作 three.js 对象
- 直接拼装复杂业务数据

### 3.5 Runtime Event Bus

定位：运行时事件总线层。

建议用于松耦合传递以下事件：

- 模型加载完成
- 对象选中变化
- 点位点击
- 摄像头点击
- 测量结果更新
- 实时数据更新
- 脚本动作触发

说明：

- 事件总线只传递事件，不保存业务状态
- 状态仍以 store 为主，避免双写

## 5. 页面拆分建议

[fullscreen.vue](e:/项目/数字交付/DigitalDelivery/src/views/visualization/viewer3d/fullscreen.vue#L1) 后续建议保留为总装页面，只负责：

- 布局容器
- `ThreeModelViewer` 挂载
- 面板和工具栏组织
- 路由参数接入
- 运行时初始化和销毁

建议从该文件中逐步抽离以下模块。

### 第一批抽离

- 顶栏
- 底部工具栏
- 剖切面板
- 对象属性面板

原因：

- 这部分边界清晰
- 逻辑依赖较少
- 可以先降低页面模板复杂度

### 第二批抽离

- 导航面板
- 结构树面板
- 设备列表面板
- 场景方案面板

原因：

- 都属于右侧侧栏业务视图
- 抽离后页面状态管理会更清晰

### 第三批抽离

- 构件属性右键菜单与编辑弹窗
- 点位编辑面板
- 摄像头视频面板
- 测量结果面板

原因：

- 这些能力会在后续阶段持续增长
- 必须提前从主页面剥离

## 6. ViewerAdapter 设计

建议建立统一适配器，封装现有 `ThreeModelViewer` 暴露方法。

建议接口分类如下。

### 模型与场景

- `loadModel`
- `resetView`
- `setQuality`
- `toggleProjection`
- `setPresetView`

### 选择与聚焦

- `selectObjectByUUID`
- `clearSelection`
- `highlightObjectByUUID`
- `focusObjectByUUID`
- `focusObjectsByUUIDs`
- `getSelectedObject`
- `getSceneTree`

### 可见性与隔离

- `isolateObjectByUUID`
- `showOnlyUUIDs`
- `filterVisibleUUIDs`
- `clearIsolation`
- `setMeshOpacityByUUID`

### 工具

- `setInteractionMode`
- `toggleFirstPerson`
- `setClippingEnabled`
- `setClippingRange`
- `clearMeasurements`
- `getMeasurements`

### 媒体与标记

- `setAnchors`
- `setCameraAnchors`
- `setLinkedPoints`
- `setLinkedPointsVisible`
- `openCameraOverlay`
- `closeCameraOverlay`

说明：

- 当前底层尚未具备 `setAnchors`、`setCameraAnchors` 这类接口，后续应增补到内核层
- 页面只依赖适配器，不直接面向 viewer 组件实例

## 8. 运行时初始化流程

建议初始化顺序：

1. 读取路由参数与模型信息
2. 初始化 runtime store
3. 挂载 `ThreeModelViewer`
4. 创建 `ViewerAdapter`
5. 绑定 viewer 事件
6. 加载工程包或场景方案
7. 恢复显示模式、点位、摄像头、测量状态
8. 开启实时数据和脚本服务

销毁顺序：

1. 停止实时数据订阅
2. 停止脚本执行
3. 清理视频播放器实例
4. 清理运行时事件总线订阅
5. 销毁 viewer 绑定和临时状态

## 10. 阶段一实施任务拆分

建议按以下顺序实施。

### 任务 1：抽出运行时 store

目标：

- 从 `fullscreen.vue` 提取运行态与工程态状态

输出：

- `viewer3dRuntime.js`
- `viewer3dProject.js`

### 任务 2：建立 ViewerAdapter

目标：

- 统一封装 viewer 调用

输出：

- `viewerAdapter.js`
- `useViewerRuntime.js`

### 任务 3：拆分基础 UI 组件

目标：

- 降低 `fullscreen.vue` 模板复杂度

输出：

- 顶栏组件
- 底部工具栏组件
- 剖切面板组件
- 对象属性面板组件

### 任务 4：拆分右侧侧栏

目标：

- 将导航、结构、设备、场景方案拆为可维护面板

输出：

- `NavigationPanel.vue`
- `StructurePanel.vue`
- `DevicePanel.vue`
- `SceneSchemePanel.vue`

### 任务 5：整理运行时服务层

目标：

- 将定位、隔离、显示模式、场景方案应用等能力从页面文件中迁移出去

输出：

- `selectionService.js`
- `visibilityService.js`
- `schemeService.js`

## 11. 阶段一验收标准

满足以下条件可视为架构拆分完成。

- `fullscreen.vue` 不再包含大段业务逻辑函数
- 页面主要职责缩减为布局与初始化
- viewer 操作通过统一适配器完成
- 工具栏和主要面板均为独立组件
- 关键功能与拆分前行为保持一致
- 后续新增点位、摄像头、测量能力时无需回到主页面中堆逻辑

## 12. 风险与控制建议

主要风险：

- 拆分过程中功能回归
- store 与 service 职责边界不清
- 适配层设计过薄，仍挡不住页面直接访问 viewer
- 组件拆分后 props 与事件链过长

控制建议：

- 拆分时保持功能逐批迁移，不一次性重写
- store 只保存状态，复杂流程放 service
- 页面禁止再新增直接 `viewerRef.value.xxx`
- 复杂共享能力优先使用 composable 或 service，不用层层透传
