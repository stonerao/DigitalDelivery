# 三维全屏场景需求符合性核查清单

核查时间：2026-04-24

核查范围：`/visualization/3d-fullscreen` 当前仓库实现、运行时界面、相关设计文档与示例数据。

说明：

- 本核查基于当前代码仓库与文档证据进行判断。
- 本核查不等同于现场模型资产实物验收；若缺少工程包、场景资源清单或现场模型产物，仅能判定“无法证明满足”。

## 一、总体结论

当前三维全屏场景已经具备较完整的三维浏览与巡视平台能力，包括：

- 三维模型加载与实时渲染
- 第一人称漫游
- 结构树/系统/KKS 快速定位
- 点位、摄像头、测量、剖切、书签
- 业务对象绑定、文件绑定、部分实时数据联动

但如果按照需求文档中“全厂级漫游展示、导航图、主辅机完整参数与台账展示、扫描建模定期更新”的口径验收，当前状态只能判定为：

- 第 1 条：部分满足
- 第 2 条：部分满足
- 第 3 条：基本满足
- 第 4 条：不满足

结论：当前实现更接近“具备三维巡视平台能力的场景系统”，但尚不能直接认定为“已满足全厂级三维场景需求”。

## 二、需求逐条核查

| 需求编号 | 需求摘要 | 结论 | 说明 |
| --- | --- | --- | --- |
| 1 | 全厂场景漫游，虚拟还原厂区环境、主辅机、管道阀门仪表，并展示主机及重要辅机参数和台账 | 部分满足 | 已具备三维场景漫游、对象选择、业务绑定、文件绑定、测点与摄像头能力，但缺少全厂资产覆盖证明，fullscreen 也未完整展示设备参数和台账卡片 |
| 2 | 配置可瞬间定位的导航图，支持游走巡视、查看、快速定位 | 部分满足 | 已支持第一人称漫游、树形导航、系统/KKS 快速定位、书签，但未见图形化导航图/小地图 |
| 3 | 从二维升级为可交互巡视的三维可视化界面，并实现实时高清渲染 | 基本满足 | 三维查看、渲染、画质档位、LOD/资源清单消费、剖切、测量等能力已具备 |

## 三、当前已具备能力

### 1. 三维巡视与交互能力

已具备以下能力：

- GLTF / GLB / IFC / OBJ 模型加载
- 旋转、平移、缩放、正交/透视切换
- 第一人称漫游
- 预设视角、书签视角
- 对象拾取、高亮、聚焦、隔离
- 剖切与剖切动画
- 测量、截图

说明：这些能力已经能支撑“进入三维场景后进行巡视、查看、定位”的基本使用。

### 2. 场景联动与业务绑定能力

已具备以下能力：

- 场景树构建与模型内构件导航
- 对象与 KKS 绑定
- 对象与文档绑定
- 测点关联
- 摄像头点位与视频弹窗
- 运行时事件、实时数据、后台桥接的基础接入能力

说明：这部分说明平台已经具备“设备对象与业务数据建立关系”的底座。

### 3. 运行时性能与展示能力

已具备以下能力：

- 高/中/低/平面渲染画质切换
- 质量档映射
- `scene-manifest` 运行时消费
- 弱机降级
- 顶视图平面渲染
- 若干运行时性能优化能力

说明：这部分基本满足“三维场景实时渲染”的需求方向。

## 四、明确差距清单

### 差距 1：缺少图形化导航图

现状：

- 当前提供的是导航树、系统定位、KKS 定位、视角书签。
- 当前没有看到厂区平面导航图、小地图、鸟瞰导航图之类的图形化导航能力。

影响：

- 无法满足需求中“配置可以瞬间定位的导航图”的表述。
- 当前定位方式偏“树与文本检索”，不是“图形化瞬时定位”。

整改建议：

1. 增加厂区导航图组件，支持点击区域跳转。
2. 导航图与模型区域、楼栋、系统节点、书签视角建立映射。
3. 支持“当前位置”标识、视角同步、区域高亮。

### 差距 2：fullscreen 未完整展示主辅机参数与台账

现状：

- fullscreen 当前对象信息面板主要显示：
  - KKS 编码
  - 名称
  - 类型
  - 状态
  - 系统信息
  - 绑定文件
  - 测点关联
- 但未展示：
  - 设备台账号
  - 制造厂家
  - 设备型号
  - 额定功率/电压
  - 结构参数
  - 运行参数

影响：

- 不能满足需求中“同时显示主机和重要辅机设备参数，包括台账信息”的验收口径。

整改建议：

1. 将普通页已有的设备资料卡能力迁入 fullscreen。
2. 在 fullscreen 对象面板中补齐：
   - 台账基础信息
   - 结构参数
   - 运行参数
   - 数据来源与更新时间
3. 区分演示数据与真实实时数据，避免验收歧义。

### 差距 3：无法证明当前场景已覆盖全厂资产范围

现状：

- 需求中要求覆盖：
  - 办公楼
  - 厂房
  - 道路
  - 绿植
  - 路灯
  - 主辅机设备
  - 管道
  - 阀门仪表
  - 宣传栏
  - 雕像装饰
- 当前仓库没有检出一份可以直接核对的“当前场景资产覆盖清单”或完整工程包资源目录。
- 示例文档和演示数据更偏“汽机房模型”“锅炉系统”“控制室”“冷却塔”等局部区域。

影响：

- 无法证明当前交付的是“全厂级场景”，只能证明具备“场景展示平台能力”。

整改建议：

1. 补一份全厂资产覆盖清单。
2. 建立“资产类型 -> 场景节点/模型 -> 是否已入模 -> 是否可定位 -> 是否绑定业务数据”的验收矩阵。
3. 对缺失资产逐项标记为：
   - 已建模
   - 已入场景
   - 已绑定
   - 未完成

### 差距 4：当前更像局部区域场景，不是全厂漫游场景

现状：

- 接口设计样例以“汽机房模型”为主。
- 联动与示例数据主要集中在：
  - 锅炉系统
  - 给水泵系统
  - DCS 控制系统
  - 控制室
  - 冷却塔

影响：

- 从现有证据看，当前更像“单模型/单区域/局部系统场景”，而不是“完整全厂场景”。

整改建议：

1. 明确项目范围是“局部区域示范”还是“全厂级交付”。
2. 若目标是全厂级，需要补厂区总图、建筑群与全厂级导航索引。
3. 若当前阶段只是局部示范，应在需求说明和验收口径中明确阶段边界。

### 差距 5：扫描建模定期更新能力未落地

现状：

- 当前已有：
  - `scene-manifest.json`
  - 质量档
  - LOD 运行时消费
  - 资源分组
- 当前未完成：
  - 材质整理与贴图标准化
  - 贴图烘焙
  - 模型分组、LOD、压缩、减面离线生成
  - 不同质量档产物生成
  - 完整发布包与资源目录规范
  - 按需加载与延迟加载

影响：

- 无法形成“现场扫描 -> 模型加工 -> 资源发布 -> 场景更新”的闭环。
- 因此不能满足“根据现场建设进度定期扫描、建模更新”的要求。

整改建议：

1. 建设离线资产处理流水线。
2. 将扫描模型更新流程标准化为：
   - 扫描数据输入
   - 模型清洗/整理
   - 贴图处理
   - LOD/压缩/减面
   - `scene-manifest` 生成
   - 工程包发布
3. 建立版本管理与变更记录。

### 差距 6：厂内状态展示仍属于部分满足

现状：

- 已有 KKS、测点、运行参数、实时数据接入基础能力。
- 但从文档与示例数据看，部分数据仍是演示性质，且若干接入协议与状态同步能力未完成。

影响：

- 当前能展示“部分设备状态”和“部分运行参数”，但不能直接认定已实现“全面掌握厂内状态”。

整改建议：

1. 明确哪些参数为真实实时数据，哪些为演示数据。
2. 建立主机、重要辅机、关键测点的最低展示标准。
3. 补齐状态接入、更新时间、异常标识与数据质量标识。

## 六、整改优先级建议

### P0

- 补 fullscreen 的设备参数与台账展示
- 明确当前项目范围是“局部区域”还是“全厂级”
- 补全厂资产覆盖清单

### P1

- 增加图形化导航图/小地图
- 打通主机与重要辅机真实参数展示
- 建立导航图与区域/系统/视角映射

### P2

- 建设离线资产处理流水线
- 建立扫描建模更新闭环
- 完善版本管理、发布包与资源目录规范

## 七、最终判断

如果按“平台能力”评估，当前实现已经具备较强的三维巡视与场景交互基础。

如果按“需求文档最终交付口径”评估，当前不能判定为完全满足，主要缺口集中在：

1. 导航图缺失
2. fullscreen 参数/台账展示不足
3. 全厂资产覆盖证据不足
4. 当前场景更偏局部区域
5. 扫描建模定期更新链路未落地

因此，建议当前对外口径使用：

`当前版本已完成三维巡视平台能力建设与局部场景展示，距离全厂级完整交付仍有明确差距项待补齐。`
